Loyalty & VIP Tier Program Architecture

How the Shopify storefront, Shopify Flow automation, and the custom Customer API work together to power loyalty points, VIP tiers, and benefits.

1Context

VIP loyalty program allows customers accumulate loyalty points (based on lifetime spend) and are being placed into VIP tiers based on their points. Each tier unlocks benefits (discount codes, birthday gifts, curated products). Shopify alone cannot express this logic, so the program is built from three cooperating systems:

System Role Repository
Shopify (Plus) System of record — customers, orders, metafields, metaobjects, tags (platform)
Shopify Flow No-code automation — recalculates points & re-assigns tiers when an order is created (managed in Shopify Admin)
Customer API Node/Express proxy — secure reads/writes the Storefront cannot do (enrollment, profile, wishlist) GitHub repository
Liquid theme Storefront — renders tier status, gates content per tier this repo

The purpose of this ADR is to give the client a single, shared picture of how the pieces fit together, what data they share, and where the dependencies (and risks) lie.

2Data Model (ERD)

All loyalty state lives inside Shopify — there is no external database. The entities below are Shopify-native objects (Customer, Order) plus custom metafields, metaobjects, and tags.

Key fields

  • customer.metafields.custom.loyalty_points — accumulated points (≈ lifetime spend).
  • customer.metafields.custom.loyalty_enrolled — enrollment timestamp; its presence means "already enrolled" and prevents re-enrollment.
  • customer.tags["loyalty_vip_tier:<tier_id>"] — the customer's current tier. This tag is the single value both Flow and the theme read to gate experiences.
  • vip_tier metaobjects — the tier catalog (name, points_condition threshold, and the list of benefit metaobjects). Edited in Shopify Admin.

3System Architecture & Dependencies

Dependency summary

  • The theme depends on the Customer API (runtime fetch) and on Shopify Liquid objects (customer.tags, customer.metafields, shop.metaobjects.vip_tier).
  • The Customer API depends on Shopify Admin API (read/write_customers) and Storefront API (metaobjects), plus a deployment host (Vercel) and env secrets (*_ADMIN_API_TOKEN, SHOPIFY_STOREFRONT_ACCESS_TOKEN, ID_TOKEN_KEY_B64).
  • Shopify Flow depends on order events and the same customer metafields/tags.
  • Everything ultimately depends on Shopify as the single source of truth — no separate loyalty database exists.

4The Two Write Paths

Loyalty state (loyalty_points + loyalty_vip_tier tag) can be written by two independent actors. This is the most important thing for the client to understand.

4a. Shopify Flow — recurring recalculation (event-driven)

Runs every time an order is created. It (1) recomputes loyalty_points, then (2) decides the customer's tier purely from amount spent, and (3) — if the customer moved up — tags them loyalty_tier_upgraded (an upgrade signal, e.g. for a congratulations email).

Tier Tag applied Condition (amount spent)
Tier 3 loyalty_vip_tier:tier_3 ≥ 6000
Tier 2 loyalty_vip_tier:tier_2 ≥ 2400 and < 6000
Tier 1 loyalty_vip_tier:tier_1 < 2400

When a tier is assigned, the other two tier tags are removed, so a customer always holds exactly one loyalty_vip_tier:* tag.

A 1-minute Wait precedes each tier branch so the "currently tier_X?" comparison reads the customer's existing tag before it is overwritten — that is how the flow distinguishes an upgrade from a no-op.

4b. Customer API — first-time enrollment (on-demand)

Runs when an un-enrolled customer first loads their account page and the theme calls GET /customer.

✅ Benefits of this design
  • No extra database to operate — Shopify remains the single source of truth.
  • The API keeps Admin API tokens off the storefront (security); CORS limits callers to the Shopify store domains.
  • Tiers and benefits are merchant-editable via vip_tier metaobjects in Shopify Admin — no code deploy needed to change names, thresholds, or benefits.
