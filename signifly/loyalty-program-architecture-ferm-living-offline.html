1Context: what we are building

Ferm Living is adding a tiered loyalty program: members earn a tier from their purchase behaviour, and each tier unlocks a set of benefits. This document is a reference for the integration: the systems involved, how data flows between them, and how tiers are computed and delivered. It is written to stay useful over time: the specific tiers, thresholds and benefits will change, but the machinery described here does not.

One property shapes the whole design: tiers depend on both spend and frequency, weighted toward recent activity, so a member's tier can rise with recent orders and decay when they go quiet. That rolling-window requirement is the reason the engine is built the way it is (see §5). Shopify cannot express it alone, so the program is assembled from a few cooperating systems, each with a clear owner:

A custom Shopify app that owns the tier logic and the per-customer timer: on each order it evaluates that customer and upgrades on the spot, sets a re-check for when the window lapses, and downgrades then if warranted. It also brokers the storefront operations the theme cannot do directly (enrolment, profile, wishlist) and backfills existing customers. One app, installed on Ferm Living's market stores. Signifly

ℹ︎ Net-new vs. existing

Today the theme is a pure Shopify + Liquid storefront with third-party scripts (Clerk, Klaviyo, Swym, …) and no custom backend. The loyalty program adds net-new customer metafields, tier metaobjects, Klaviyo flows, and one Shopify app (the Loyalty app) that runs the tier engine and the secure customer operations. Because Ferm Living's market stores share one Plus organisation, a single custom app installs across them all. Its database holds only plumbing, per-store tokens and the re-check timers, never a copy of the tier: Shopify stays the source of truth for a customer's tier.

3Data model: where loyalty state lives

A customer's tier lives inside Shopify, on the customer as a tag plus metafields; that is the source of truth the storefront and Klaviyo read. The only state the app keeps in its own database is operational: per-store tokens and the per-customer re-check timer. Tier thresholds and benefits live in net-new metaobjects. (Customer metafields are new for this theme; today only product/variant metafields exist.)

Key fields

  • customer.tags["loyalty_vip_tier:<core|curator|collector>"] is the customer's current tier. This single tag is what the theme reads to gate experiences and what Klaviyo syncs. A customer always holds exactly one tier tag.
  • customer.tags["loyalty_enrolled"]: presence means “already a member”; set on the first purchase and prevents re-enrollment / a repeated welcome.
  • customer.metafields.loyalty.* holds the audit & control fields the app stamps: enrolled_at, tier_changed_at, last_evaluated_at. These make tier changes traceable and feed analytics (the tier itself is the tag above).
  • The app's database (we suggest a managed Postgres such as Supabase) holds the re-check timer: (shop, customer_id, recheck_at, status), indexed on recheck_at. The daily sweep reads only the due rows. The tier itself is never copied here; it stays on the customer in Shopify.
  • vip_tier metaobjects: the merchant-editable tier catalog: display name, benefit list, and per-market thresholds. Editing names, numbers or benefits in Admin needs no code deploy.
  • Order history, read by the app via the Admin API. To resolve a tier it counts a customer's orders inside the window (filtered by created_at) alongside the lifetime amountSpent. This trailing-window read is the part a backend does that Shopify Flow cannot (see §5).

4System architecture & dependencies

The map splits into two paths that meet at Shopify's data: writing a tier (the engine) and reading it (the storefront).

Dependency summary

  • The Loyalty app (the engine) depends on the orders/create webhook to fire, the Admin API (read orders by date, read/write customer tags + metafields, read the vip_tier metaobjects), and its own database for the re-check timers. It writes the tier and signals Klaviyo.
  • The theme depends on the app for the interactive bits (account, enrol, wishlist) through the App Proxy and a theme app extension, and on Shopify Liquid objects for rendering (customer.tags, customer.metafields, shop.metaobjects.vip_tier). Tier status renders from the tag with no app call, so it stays fast and cacheable.
  • The app depends on a per-store OAuth token (held in its DB), the Admin and Storefront APIs, and its host. It installs on each store via OAuth; because Ferm Living's market stores share one Plus organisation, a single custom app covers them all.
  • Klaviyo depends on the tier reaching it (the Shopify tag syncs to a profile property, or the app sends an explicit tier-change event) to fire benefit flows. Klaviyo does not own tier state.
  • Swym depends on the customer and theme for the wishlist; tier-aware perks read the same tier tag.
  • Everything ultimately depends on Shopify as the source of truth for the tier. The app's database holds only per-store tokens and the re-check timers, never a copy of the tier.

5The tier engine: the Loyalty app

All the tier logic lives in the Loyalty app. It runs per customer, on an event: an order is placed, the orders/create webhook hits the app, and it evaluates that one customer (recent orders plus lifetime spend against the thresholds), upgrading on the spot. It records a re-check timer for when the window lapses; a daily sweep re-checks only the customers whose timer is due and downgrades those whose window passed. Nothing scans the ~30k base, because nothing needs to: only someone who just ordered can move up, and only a customer whose window is lapsing can move down.

ℹ︎ Why the app owns the timer, not Flow

The intuitive design is “wait the window after an order, then re-check.” The only question is where the wait lives, and two Shopify Flow limits push it into the app: Flow's Wait caps at 90 days (the windows here are 180 to 365), and Flow conditions can read only lifetime spend and order counts, never “orders in the last N days.”

The app has neither limit: it queries a customer's recent orders directly, and it holds the re-check as a recheck_at timestamp in its own database, so the wait has no ceiling. Shopify Flow stays optional, useful only as a no-code hook (the app can expose a custom Flow action).

How it works

  1. Order placed, evaluate that one customer. The orders/create webhook calls the app. It reads the customer's recent orders (Admin API, filtered by created_at) and lifetime spend, resolves their highest qualifying tier from the vip_tier metaobjects, and if it is higher than now, writes the new tier immediately, enrolling a first-time buyer.
  2. Record the re-check timer. It upserts recheck_at = order_date + window in its database, one row per customer, a new order just overwrites it. No long-lived in-flight job, only a timestamp.
  3. A daily sweep re-checks only the due set. Once a day the app queries the rows where recheck_at <= now and re-evaluates each of those customers. A debounce (has a newer qualifying order arrived?) keeps the latest order in charge of the clock; if the window lapsed, it downgrades to the customer's spend-floor tier. That is tens to low hundreds a day, never the whole base.
  4. Delivery runs through Klaviyo. On any tier change the app sends a tier-change event (or the tag syncs as a profile property), and Klaviyo fires the matching flow: welcome, congratulations, win-back, birthday, early-access. Tier stays one-way Shopify to Klaviyo, so a sync never overwrites it.

The evaluation is the same few lines on both paths; the thresholds come from the metaobjects, so they stay configuration:

// on the orders/create webhook, and again in the daily sweep
const orders = await admin.orders(customerId, { since: now.minus(windowDays) });
const tier   = highestQualifyingTier({
  lifetimeSpent: customer.amountSpent,   // all-time
  recentOrders:  orders.length,          // within the window
}, tierRules);                           // tierRules = vip_tier metaobjects

if (tier !== current) writeTier(customerId, tier);     // tag + metafields (Shopify)
db.upsertRecheck(customerId, now.plus(windowDays));    // the per-customer timer (app DB)
if (tier !== current) notifyKlaviyo(customerId, { from: current, to: tier });

The app's second job: secure storefront operations

The same app brokers the customer operations a Liquid storefront cannot do safely, because the Storefront API cannot write customer metafields and Admin tokens must never reach the browser. The storefront reaches it through the App Proxy (calls under the store's own domain, HMAC-signed and scoped to the logged-in customer):

  • Enrolment & loyalty reads: the customer's tier, benefits and status, plus the tier catalog from the metaobjects.
  • Profile: register, update name / preferences / sizes, change password, read the full profile.
  • Addresses: add, update, delete.
  • Backfill: existing customers who never trigger an order webhook get an initial tier from a one-off pass over their order history.

The storefront UI ships as a theme app extension (app blocks and an app embed), so the tier UI drops into the theme through the theme editor with no per-store Liquid edits.

One app, every market

It is a custom Shopify app. Ferm Living's market stores (.com, DK, UK, US, SE) all sit in one Plus organisation, so a single custom app installs across them via OAuth (a per-store token kept in the app's database), auto-registers the orders/create webhook, and carries the App Proxy and theme extension with it. A custom app within one Plus org gets protected-customer-data access without review; reading order history older than 60 days uses the read_all_orders scope, which the tier maths needs.
